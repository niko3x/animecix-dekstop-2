    # Release Runbook — AnimeciX Desktop v2

    Follow these steps for every release. A release takes ~20–30 minutes of active time + ~15 minutes of CI wait.

    ## 0. Prerequisites (once per release)

    - [ ] Complete `PRE-SHIP-SECURITY-CHECKLIST.md` (first release only — re-read every time).
    - [ ] Confirm `main` is green (CI passing on latest commit).
    - [ ] Confirm no in-flight downloads or caches in your local dev app — avoid accidental lock files.

    ## 1. Bump version

    ```bash
    cd animecix-v2
    # Choose one: patch / minor / major
    npm version patch --no-git-tag-version
    git add package.json package-lock.json
    git commit -m "chore: bump version to v$(node -p "require('./package.json').version")"
    git push origin main
    ```

    ## 2. Tag the release

    ```bash
    VERSION=$(node -p "require('./package.json').version" --cwd animecix-v2)
    git tag -a "v${VERSION}" -m "Release v${VERSION}"
    git push origin "v${VERSION}"
    ```

    The tag push triggers `.github/workflows/release.yml`.

    ## 3. Watch CI

    - Visit Actions tab → find the Release workflow run.
    - Expect both `macos-14` and `windows-latest` jobs to complete in 10–15 min each.
    - If `verify-tag` fails → your tag doesn't match `package.json`. Delete the tag (`git tag -d v${VERSION} && git push origin :refs/tags/v${VERSION}`) and restart from step 1.
    - If signing hangs on macOS → confirm `-T /usr/bin/codesign` flag is still in the workflow.
    - If notarization fails with "Invalid credentials" → ASC API key rotated; update `APPLE_API_KEY` secret.

    ## 4. Verify draft release artifacts

    Visit https://github.com/CaptainSP/AnimeciX-Desktop-Apps/releases → the new draft release. Confirm these 7 assets are attached:

    - `AnimeciX-<ver>-universal-mac.zip` (auto-update channel)
    - `AnimeciX-<ver>.dmg` (macOS first-install)
    - `latest-mac.yml` (manifest for macOS updates)
    - `AnimeciX-<ver> Setup.exe` (Windows installer)
    - `AnimeciX-<ver>-full.nupkg` (Windows delta source)
    - `RELEASES` (Squirrel index)
    - `latest.yml` (manifest for Windows updates)

    If any missing → re-run the workflow (rerun the failed job via GitHub UI).

    ## 5. Smoke test on clean machines

    - [ ] **macOS**: Download the DMG on a clean macOS 14+ VM or second machine with no dev tools. Drag to Applications. Launch. Verify:
          - No Gatekeeper "unknown developer" warning past the first "are you sure you want to open" prompt
          - Running `spctl --assess --type execute --verbose "/Applications/AnimeciX.app"` prints `accepted; source=Notarized Developer ID`
          - App loads animecix.tv successfully
    - [ ] **Windows**: Run `AnimeciX Setup.exe` on a clean Windows 10/11 VM. Verify:
          - "Windows protected your PC" SmartScreen warning appears
          - Clicking "More info → Run anyway" installs successfully
          - App appears in Start menu, launches, loads animecix.tv

    ## 6. Edit release notes

    In the draft release on GitHub:
    - Review the auto-generated changelog (enabled via `generateReleaseNotes: true`).
    - Trim or rewrite as needed. Add migration notes if any breaking changes.
    - Confirm the version in the release title matches the tag.

    ## 7. Publish

    Click **Publish release** in the GitHub UI. This:
    - Flips `draft: false` → users' installed apps can now see the release.
    - Within 30 seconds to 4 hours, running instances of the previous version will call `checkForUpdates` and start downloading.

    ## 8. Post-publish verification

    - [ ] On a machine with the previous version installed, wait up to 4 hours (or force a check via the tray menu's "Güncellemeleri kontrol et" item). Confirm the Turkish banner `Yeni sürüm hazır` appears.
    - [ ] Click `Şimdi yeniden başlat`. Confirm the app restarts as the new version.

    ## 9. Monitor for 24 hours

    - Check GitHub Issues for update-related bug reports.
    - If a critical regression is found and the draft-release path was missed: tag a patch release (`v${VERSION+patch}`), let CI produce a new draft, publish. electron-updater will pick up the newer version on the next check cycle.

    ## Rollback

    There is no true rollback — once published, electron-updater delivers to users. To undo a bad release:
    1. Immediately "Unpublish" the GitHub Release (keeps artifacts but removes from the updater feed).
    2. Tag a patch version that reverts the bad commit.
    3. Let the patch release ship normally.
    ```
